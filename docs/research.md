# research.md

## 1. 调研目标与背景

本文档用于评估并指导一个参考 OpenClaw 架构、自建本地 host 的新项目落地方案。这里的“参考 OpenClaw”含义是：学习其渠道接入、消息路由、会话管理、AI 运行时解耦、回复分发等架构思想，而不是直接复制 OpenClaw 源码实现。

你的当前条件如下：

- 已有一套基于 OpenClaw 的源码解析文档，可作为架构参考。
- 计划在本地新建项目，路径为 `E:\Develop\ai-host-proto`。
- 模型处理层希望按 `Codex runtime` 理解，底层能力来自 Azure Foundry 上部署的 `gpt-5.4`，因此系统设计不应依赖某个公开模型名。
- 交互渠道最初希望通过微信接入，但你也接受“如果微信不合适，可改走飞书”的路线。

本文重点回答三个问题：

1. 在你当前条件下，这个想法能不能做。
2. 如果能做，第一阶段最应该怎么落地。
3. 哪些地方可以借鉴 OpenClaw，哪些地方不要在第一版就照着全量复刻。

---

## 2. 结论先行

### 2.1 总体结论

结论是：**可以实现，而且很适合采用“参考 OpenClaw 架构、重新实现一个更轻量 host”的方式来做。**

但从工程角度看，建议把目标拆成两个层次：

- **第一层：做一个通用 host 核心。**
  - 负责统一接入渠道消息。
  - 统一会话、路由、上下文、日志和回复分发。
  - 对上游渠道和下游模型运行时保持解耦。
- **第二层：按渠道逐步接入。**
  - 第一优先级建议是飞书。
  - 个人微信只建议作为实验方案，而不是主方案。
  - 如果未来必须走微信系并且需要长期稳定，则优先转向企业微信。

### 2.2 推荐决策

**当前默认路线建议如下：**

1. **先做通用 host 核心。**
2. **第一条正式渠道先接飞书。**
3. **个人微信保留为实验适配器，不作为核心依赖。**
4. **模型层抽象成 `Codex runtime`，不要把业务层写死到具体模型名。**
5. **第一版只做最小闭环，不追求 OpenClaw 的全量能力。**

这是最稳、最省时间、也最适合后续扩展的路线。

## 3. OpenClaw 中值得借鉴的架构

结合当前仓库中的解析文档，OpenClaw 最值得借鉴的不是某个具体模块，而是它对“多渠道 AI host”的整体拆分方式。

### 3.1 渠道插件化思路值得保留

从 `channels.md` 可以提炼出一个核心思想：**每个消息平台只是一个适配器，Gateway 不应该知道平台的具体 API 细节。**

你后续的新项目也应保持这一点：

- Telegram、飞书、企业微信、个人微信，不应该直接写进业务主流程。
- 不同平台只负责：
  - 接收入站消息。
  - 将平台消息转换为统一结构。
  - 执行出站发送、编辑、删除等动作。
- 主流程只处理统一后的消息上下文。

这会直接带来几个好处：

- 后续更换渠道不会重写核心逻辑。
- 可以先做飞书，再补微信，而不是二选一。
- 可以将“个人微信实验能力”和“正式渠道能力”分开维护，降低风险。

### 3.2 统一 `MsgContext` 是关键设计

从 `auto_reply.md` 与 `routing.md` 可以看到，OpenClaw 的主流程并不是围绕某个平台消息对象展开，而是围绕统一的 `MsgContext` 展开。

这点非常值得保留。

在你的新项目中，建议定义统一消息上下文，至少包含：

- `channelId`：渠道标识，如 `feishu`、`wechat-personal`、`wecom`。
- `accountId`：对应机器人或登录账号。
- `peerId`：对端标识，可以是用户、群、会话、线程。
- `sessionKey`：会话主键。
- `messageId`：原始消息 ID。
- `senderId`、`senderName`。
- `text`：用户文本。
- `rawPayload`：保留原始平台事件，便于排障。
- `isGroup`、`mentions`、`threadId` 等扩展字段。

这样做的意义在于：

- 你的 AgentRuntime 不需要知道消息来自飞书还是微信。
- 路由规则与会话持久化可以复用。
- 命令系统、日志系统、审计系统可以统一实现。

### 3.3 路由与 `SessionKey` 设计值得直接参考

`routing.md` 里最有价值的一点，是把会话从“平台原始事件”中抽离出来，形成可预测的 `SessionKey`。

建议你的系统也采用同样的思路：

- `sessionKey` 应由多个稳定维度组合生成，而不是直接拿平台对话 ID。
- 推荐至少包含：
  - `agentId`
  - `channelId`
  - `accountId`
  - `peerId`
  - 必要时包含 `threadId`

例如：

```txt
agent:default:feishu:bot-main:user-12345
agent:default:wechat-personal:wx-main:room-9988
```

这样设计的好处：

- 同一个人从不同渠道来消息，不会串会话。
- 多账号并行时不会混历史。
- 未来支持多 Agent 时能平滑升级。
- 做会话锁、并发控制、清理历史时都更容易。

### 3.4 Reply Dispatcher 的思想应保留，但实现可简化

OpenClaw 的回复分发器负责“流式输出、缓冲、typing、节流、完成态控制”。

这套思想值得学习，但第一版不要做得太重。

你只需要保留三件事：

1. **把模型流式输出切成可发送的块。**
2. **做最基本的 debounce / flush。**
3. **针对不同平台决定是编辑原消息还是连续发多条。**

第一版不要一开始做：

- 复杂的跨平台 typing 抽象。
- 太细的 token 级别输出控制。
- 平台能力过于统一的“完美抽象”。

因为不同平台差异非常大：

- 飞书与企业微信更容易走机器人 API。
- 个人微信大概率只能走桌面自动化，编辑消息能力弱，稳定性更差。
- 有的平台支持编辑已有消息，有的平台只适合分段发送。

所以建议：**保留 dispatcher 的角色，但接受平台差异。**

### 3.5 Agent Runtime 与渠道解耦是最重要的核心之一

从 `agents.md` 与 `auto_reply.md` 可以提炼出 OpenClaw 的另一条主线：

- 渠道层负责收消息和回消息。
- Agent 运行时负责读取历史、构造输入、调用模型、处理工具结果、输出流式文本。

你的新项目也应坚持这一点。

建议把模型处理层抽象成统一接口，例如：

```ts
interface AgentRuntime {
  run(input: RuntimeInput): Promise<RuntimeResult>
}
```

其中 `RuntimeInput` 至少包括：

- `sessionKey`
- `messages`
- `systemPrompt`
- `tools`
- `stream`
- `metadata`

`RuntimeResult` 至少包括：

- `finalText`
- `usage`
- `toolCalls`
- `finishReason`

如果你当前希望把信息处理层视为 Codex，而不是直接面向底层模型，那么这里就应该做成 `CodexRuntimeAdapter`。

也就是说：

- host 核心不关心底层是 Azure Foundry、OpenAI、别的 provider，还是你自己的封装。
- host 核心只知道：给 runtime 一段上下文，取回回复或流式块。

这会极大提升后续扩展性。

## 4. 你的条件下的可行性判断

### 4.1 模型侧可行性

模型侧是可行的，而且相对容易。

你的关键条件不是“某个公开模型名”，而是：

- 你已经在 Azure Foundry 部署了可调用的模型能力。
- 你希望在 host 中把它抽象成 `Codex runtime`。

这意味着系统设计时不需要把“模型选择”做成第一优先级问题。第一版只需要完成：

- 配置 endpoint / deployment / auth。
- 将统一消息历史整理成 runtime 输入。
- 支持同步或流式输出。
- 记录返回文本、usage 和异常。

换句话说，模型接入在你的整体方案里不是主要风险项。

### 4.2 渠道侧可行性

渠道侧也可行，但不同渠道风险差异巨大。

#### 个人微信

个人微信可以做，但**只适合作为实验方案**。

主要原因：

- 大概率无法走稳定、长期可依赖的官方 bot API。
- 很多方案本质上是桌面自动化、RPA、消息 Hook 或非官方适配。
- 风险包括：
  - 客户端升级失效。
  - 登录态不稳定。
  - 自动化控制容易受 UI 变化影响。
  - 封号或限制风险更高。
  - 很难作为正式工程的核心链路。

如果你只是：

- 自己用；
- 小团队内部试验；
- 想先验证“在微信里和本地 host 对话”这件事；

那么个人微信路线可以保留，但应明确标注为：**实验适配器**。

#### 飞书

飞书是当前最适合第一阶段正式落地的渠道。

原因包括：

- 官方开放接口成熟。
- 机器人、Webhook、应用消息能力明确。
- 事件回调与服务端模式清晰，便于做稳定集成。
- 对“企业内部 AI 助手 / 个人工作流助手”这类场景非常友好。
- 更适合作为通用 host 的第一条正式链路。

如果目标是尽快跑通一个工程质量更高的 MVP，飞书优先级应该排第一。

#### 企业微信

企业微信适合作为未来“微信系正式版”的主路线。

它的优点是：

- 官方能力完整。
- 与企业工作流结合更自然。
- 更适合长期稳定维护。

它的不足是：

- 接入复杂度通常高于飞书。
- 配置、权限、企业管理后台、回调验证等流程更多。
- 如果当前只是验证原型，投入产出比未必高于飞书。

因此企业微信更适合作为第二阶段或第三阶段的正式扩展，而不是第一阶段默认入口。

#### 公众号

公众号可以做，但定位与 IM 式对话型 host 不完全一致。

它更像：

- 用户消息接入；
- 客服或回调式交互；
- 事件驱动消息处理。

如果你的目标是构建一个“像即时通讯机器人一样自然对话”的 host，公众号通常不是最顺手的第一选择。

### 4.3 最终推荐顺序

如果从“正式工程优先级”排序，建议如下：

```txt
飞书 > 企业微信 > 公众号 > 个人微信
```

如果从“自用实验价值”排序，则可以理解为：

```txt
飞书 ≈ 个人微信实验 > 企业微信 > 公众号
```

原因是：

- 飞书最适合尽快做出稳定 MVP。
- 个人微信最适合满足你“我就是想在微信里用”的原始诉求，但风险明显更高。
- 企业微信适合长期正式化。
- 公众号适合特定业务形态，不一定适合作为对话主渠道。

## 5. 渠道方案对比

| 渠道 | 接入方式 | 开发复杂度 | 稳定性 | 合规性 | 适合阶段 | 是否建议第一阶段 |
|---|---|---:|---:|---:|---|---|
| 个人微信 | 非官方/RPA/桌面自动化/适配框架 | 中到高 | 低 | 低 | 自用原型、实验验证 | 否 |
| 飞书 | 官方 Bot / Webhook / 应用消息 API | 中 | 高 | 高 | MVP、正式起步 | 是 |
| 企业微信 | 官方应用 / 机器人 / 回调 API | 中到高 | 高 | 高 | 正式微信系方案 | 否，建议第二阶段 |
| 公众号 | 官方消息回调 / 被动回复 | 中 | 中到高 | 高 | 客服、触发式交互 | 否 |

### 5.1 个人微信路线建议

如果坚持保留个人微信，建议这样处理：

- 单独做 `wechat-personal-adapter`。
- 将其部署在独立的 Windows 节点上。
- 只承担：
  - 收消息；
  - 发消息；
  - 做最基本的去重与状态回传。
- 所有会话状态、路由、AI 调用逻辑都放回 host-core。

不要把核心业务逻辑写进个人微信自动化进程里，否则后续替换飞书或企业微信会非常痛苦。

### 5.2 飞书路线建议

飞书建议作为第一条正式链路，目标是尽快验证：

- 统一消息上下文是否合理。
- SessionKey 是否能稳定组织历史。
- Codex runtime 适配是否顺畅。
- ReplyDispatcher 的流式/缓冲策略是否足够。

飞书跑通之后，你的 host 核心就已经成型了。

### 5.3 企业微信路线建议

企业微信建议作为“正式微信系版本”的升级路线：

- 在核心 host 已稳定后再接入。
- 直接复用已有：
  - `ChannelAdapter`
  - `MsgContext`
  - `RouteResolver`
  - `SessionStore`
  - `AgentRuntime`
  - `ReplyDispatcher`

届时主要新增的只是渠道适配器，而不是重写系统。

## 6. 推荐的新 host 架构

### 6.1 设计目标

新 host 的目标不应该是“复刻 OpenClaw 的全部功能”，而应该是：

- 做出一个你能持续演进的轻量本地 host。
- 核心上具备渠道可插拔、runtime 可替换、会话可持久化的能力。
- 第一版先服务你自己的场景，再决定是否往多用户、多工具、多平台扩。

### 6.2 建议保留的核心模块

建议第一版最小架构包含以下 6 个核心模块。

#### 1. `ChannelAdapter`

职责：

- 接收入站事件。
- 转换为统一 `MsgContext`。
- 执行出站发送。
- 按渠道能力支持编辑、删除、typing 或分段输出。

输入：

- 平台原始事件。
- Dispatcher 生成的文本块。

输出：

- 统一 `MsgContext`。
- 平台发送结果和消息 ID。

#### 2. `MsgContext`

职责：

- 作为整个消息处理链路中的统一数据载体。
- 对命令系统、日志系统、路由系统、会话系统提供稳定输入。

输入：

- 渠道原始消息。

输出：

- 被路由、存储和 runtime 调用使用的标准对象。

#### 3. `RouteResolver`

职责：

- 根据 `channelId`、`accountId`、`peerId`、`agentId` 生成 `sessionKey`。
- 决定消息进入哪个 agent 或哪个会话。
- 为未来多 agent 扩展预留空间。

#### 4. `SessionStore`

职责：

- 读写历史消息。
- 管理会话元数据。
- 提供串行化锁，避免并发写乱历史。

第一版建议：

- 优先用 SQLite 或 JSONL。
- 不急着做复杂的摘要压缩。

#### 5. `ReplyDispatcher`

职责：

- 接收 runtime 的流式输出。
- 缓冲与节流。
- 决定是否编辑已有消息或发送新消息。
- 在结束时做完成态处理。

#### 6. `AgentRuntime`

职责：

- 构造模型输入。
- 调用 `Codex runtime`。
- 处理流式输出和工具结果。
- 返回最终文本、usage 和状态。

### 6.3 推荐的处理链路

建议的主流程如下：

```txt
Inbound Event
  -> ChannelAdapter
  -> MsgContext
  -> RouteResolver
  -> SessionStore.loadHistory
  -> AgentRuntime.run
  -> ReplyDispatcher
  -> ChannelAdapter.send/edit
  -> SessionStore.appendHistory
```

这个流程有几个明显优点：

- 渠道与模型解耦。
- 能天然支持多渠道。
- 能方便插入命令系统。
- 会话管理位置清晰。
- 后续加工具调用时也有自然扩展点。

### 6.4 第一版明确不做的能力

为了避免过度设计，建议第一版不要做以下能力：

- 多 provider key 轮换。
- 完整审批系统。
- 上下文溢出压缩。
- 子 Agent / 多 Agent 编排。
- 插件市场式扩展系统。
- 控制台 UI。
- 大规模权限系统。

这些都是 OpenClaw 体系中有价值但偏重的能力，不适合作为你的 v1 起点。

## 7. 分阶段实施路线

### Phase 0：文档与接口冻结

目标：

- 固化这份研究文档。
- 输出核心接口定义草案。
- 选定第一条正式渠道。
- 明确 MVP 范围和非目标。

结果：

- 你可以在不写太多代码的前提下，先把架构边界讲清楚。

### Phase 1：先做飞书闭环

目标：

- 接入飞书事件。
- 统一为 `MsgContext`。
- 通过 `RouteResolver` 生成 `SessionKey`。
- 从 `SessionStore` 读取历史。
- 调用 `Codex runtime`。
- 用 `ReplyDispatcher` 回发消息。
- 保存历史。

这一步完成后，整个 host 的核心就已经成立。

### Phase 2：补齐可用性能力

建议补的内容：

- `/clear`、`/stop`、`/model` 三个命令。
- 错误日志与链路追踪。
- 流式输出的缓冲与限频。
- 基本重试机制。
- 简单配置加载。

### Phase 3：扩展第二渠道

这里有两种路线：

- **路线 A：加个人微信实验适配器**
  - 满足“我就是想从微信里对话”的诉求。
  - 明确定位为实验能力。
- **路线 B：加企业微信正式适配器**
  - 面向长期稳定。
  - 面向更正式的工作流使用场景。

如果你后续主要是自己用，A 可以先于 B。
如果你后续要走正式团队使用，B 应优先于 A。

### Phase 4：再考虑增强特性

到这个阶段，再考虑：

- 工具调用。
- 文件读写。
- 命令执行审批。
- 长期记忆。
- Web 管理面板。
- 多 Agent。
- 插件机制。

这些增强能力应建立在 host 核心已经稳定的前提上。

## 8. 风险与规避策略

### 8.1 个人微信风险

风险：

- 封号或风控。
- 自动化失效。
- PC 客户端升级导致适配失效。
- 长时间运行稳定性差。
- UI 自动化依赖窗口环境，部署限制更强。

规避策略：

- 明确作为实验功能，不作为唯一主链路。
- 运行在独立 Windows 节点。
- 适配器进程尽量无状态。
- 核心逻辑全部放到 host-core。
- 提前准备飞书或企业微信替代路径。

### 8.2 模型接口耦合风险

风险：

- Azure Foundry 的 deployment 名称、接口参数、认证方式与业务代码耦合过深。
- 后续想切换 provider 或更换 runtime 时，改动面过大。

规避策略：

- 将底层调用统一封装到 `AgentRuntime` 或 `CodexRuntimeAdapter` 中。
- 在业务层只暴露统一 `run()` 接口。
- 把 deployment、endpoint、credential 放入配置层，而不是业务逻辑中。

### 8.3 多渠道会话串线风险

风险：

- 同一用户在不同平台使用时，历史可能混在一起。
- 多账号并行时，回错渠道或串会话。

规避策略：

- 明确使用结构化 `SessionKey`。
- 所有历史读取和写入都以 `sessionKey` 为核心。
- Dispatcher 在出站时必须携带原始上下文中的渠道和目标信息。

### 8.4 流式回复平台差异风险

风险：

- 不同平台消息编辑能力差异大。
- 发送频率限制不同。
- 个人微信对高频编辑特别不友好。

规避策略：

- Dispatcher 不追求完全统一行为。
- 按平台能力决定“编辑原消息”还是“分段发送”。
- 用保守的 debounce 和 flush 策略作为第一版默认值。

---

## 9. MVP 建议

### 9.1 MVP 的边界

第一版只建议支持：

- 单 runtime。
- 单管理员或极少量内部用户。
- 单渠道或双渠道。
- 文本消息优先。
- 简单会话历史。
- 基本流式输出。

### 9.2 MVP 保留命令

建议只保留三个命令：

- `/clear`：清空当前会话。
- `/stop`：中止当前生成。
- `/model`：查看或切换当前模型配置。

不要一开始就扩太多命令，否则会把精力从核心链路分散掉。

### 9.3 MVP 的存储建议

建议：

- 如果你希望简单透明，先用 JSONL。
- 如果你希望后续检索和并发更稳，优先 SQLite。

我的建议偏向：

- **第一版优先 SQLite**，因为它更适合作为小型本地 host 的长期基础。
- 如果你想更快开始，也可以先用 JSONL，然后在第二阶段平滑迁移。

## 10. 后续开发建议

### 10.1 先文档驱动，再接口驱动，再代码驱动

建议开发顺序：

1. 固化本文档。
2. 写出核心接口定义草案。
3. 明确配置文件结构。
4. 实现 host-core。
5. 接入飞书。
6. 再补第二渠道。

这样比“直接开写渠道代码”更稳。

### 10.2 优先保证两层解耦

后续开发中最重要的两层解耦是：

- **渠道层 vs host-core**
- **host-core vs runtime 层**

只要这两层边界清晰，你后面无论是加微信、飞书、企业微信，还是切换底层 runtime，都会比较轻松。

### 10.3 把个人微信视为能力插件，而不是系统基础

如果你确实很想用微信，最合理的方式不是“整个系统围绕微信设计”，而是：

- 先把系统做成中性的 host。
- 再把个人微信作为一个实验性 `ChannelAdapter`。

这样，即便个人微信路线失败，你的项目也不会废掉。

### 10.4 如果未来想扩成真正的 Gateway

未来如果你希望它进一步接近 OpenClaw 这种多渠道 AI Gateway，再逐步加入：

- 多用户与权限。
- 审批系统。
- 工具调用。
- 配置热加载。
- 插件机制。
- Web 管理端。
- 多 Agent。

但这些都不该抢在第一版之前。

## 11. 最终建议总结

综合你的条件，最推荐的执行方案如下：

### 方案 A：推荐主方案

- 新项目名：`ai-host-proto`
- 第一版目标：做一个轻量的本地 AI host
- 运行时：抽象成 `Codex runtime`
- 第一条正式渠道：飞书
- 第二条渠道：企业微信或个人微信实验适配器
- 存储：SQLite 优先
- 架构风格：参考 OpenClaw 的渠道解耦、统一上下文、路由与会话、reply dispatcher

### 方案 B：如果你一定要第一时间做微信

也可以这样做：

- host-core 先照样建立
- 第一条渠道做 `wechat-personal-adapter`
- 但必须明确：
  - 它是实验能力
  - 风险高
  - 随时可能因为客户端变化而需要返工
- 同时预留飞书适配器接口，避免被微信路线绑定

### 最后一句建议

**从“能最快做出稳定成果”的角度，先做飞书；从“满足你最直接使用习惯”的角度，可以并行保留个人微信实验适配器，但不要让它成为整个项目的基础。**
